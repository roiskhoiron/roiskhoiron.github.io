# MCP Security

> *"MCP bikin agent bisa pake tools apa aja. Keren — tapi setiap tools baru adalah potensi celah keamanan baru. Kayak buka pintu rumah — makin banyak pintu, makin banyak yang perlu dikunci."*

Waktu pertama kali setup MCP (Model Context Protocol) untuk agent CodingSkuy, aku excited. Agent bisa search artikel, baca konten, generate ringkasan — semuanya lewat tools yang terintegrasi.

Tapi semakin banyak tools yang aku tambah, semakin sadar aku: **setiap tools adalah permukaan serangan baru.**

---

### Kisah Pertama: Tools Palsu yang Hampir Kepake

Suatu hari aku coba integrate tools dari sumber tidak resmi — tools pihak ketiga yang "katanya" bisa bantu analisis sentimen. Tools-nya kelihatan legit: dokumentasi rapi, contoh jelas.

Tapi sebelum pake, aku cek source code-nya. Ada baris yang mencurigakan — tools mengirim data ke endpoint yang tidak jelas.

**Bayangkan:**
Tools itu dipanggil agent → agent kirim data ke tools → tools kirim data ke attacker. Tanpa sadar.

Pelajarannya: **tidak semua tools bisa dipercaya.** Sama seperti tidak boleh asal install package dari npm tanpa cek.

---

### Kisah Kedua: Tools Tanpa Autentikasi

Aku pernah setup tools MCP untuk internal — tanpa autentikasi. Pikirku: "Kan internal aja, nggak ada yang tau endpoint-nya."

Ternyata teman iseng coba akses endpoint itu — dan berhasil. Dia bisa panggil tools tanpa izin. Beruntung dia baik dan cuma notifikasi. Tapi kalau attacker?

**Mitigasi:**
Setiap tools MCP harus punya autentikasi sekarang. Token, API key, atau OAuth. Minimal.

---

### Praktik Keamanan MCP

**1. Autentikasi dan otorisasi**
Setiap tools wajib diautentikasi. Token atau API key. Jangan biarkan tools tanpa proteksi.

**2. Enkripsi komunikasi**
Komunikasi antara agent dan tools harus HTTPS/TLS. Data dalam perjalanan bisa dicegat.

**3. Tool whitelist**
Agent cuma bisa akses tools tertentu. Bukan bebas panggil tools di internet. Di CodingSkuy, tools MCP terbatas pada: search index artikel, baca artikel tertentu, generate ringkasan. Tidak ada akses ke database user, environment variables, atau delete.

**4. Audit logging**
Setiap panggilan tools tercatat. Agent mana, tools apa, parameter apa, hasil apa. Kalau ada insiden — bisa di-trace.

**5. Rate limiting**
Batasi berapa kali agent bisa panggil tools dalam periode tertentu. Cegah abuse atau loop tak terbatas.
