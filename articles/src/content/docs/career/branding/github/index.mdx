# GitHub

> *"Dulu GitHub-ku cuma tempat fork repositori orang lain. Isinya 10 repositori, semuanya tutorial, tidak ada satupun punyaku."*

Awal belajar coding, GitHub buatku adalah tempat "simpan kode biar aman." Aku push project bootcamp, fork repository terkenal, dan lupa. Tidak ada yang orisinal. Tidak ada yang menunjukkan kemampuanku.

Sampai suatu hari — seorang rekruter minta lihat GitHub-ku. Aku buka dengan percaya diri. Dia lihat 5 menit, lalu bilang:

> *"Ini isinya proyek tutorial semua. Mana yang beneran kamu buat sendiri?"*

Aku malu. Tidak bisa jawab. Dari situ aku sadar: **GitHub bukan tempat penyimpanan — GitHub adalah portfolio kode paling autentik yang bisa kamu punya.** Dan selama ini, portfolio-ku kosong.

---

### Kisah Pertama: Malu Sampe Benerin GitHub

Setelah kejadian itu, aku putuskan untuk benar-benar merombak GitHub. Bukan nambah repositori — tapi **membangun sesuatu yang orisinal** dari nol.

![GitHub contribution graph showing green squares and profile overview](/articles/images/journey/01-github-contribution-graph.jpeg)

Aku mulai dengan bikin CodingSkuy — aplikasi Flutter untuk artikel coding Bahasa Indonesia. Ini proyek pertamaku yang benar-benar dari ide sendiri, bukan tutorial.

Langkah-langkah yang kulakukan:

**1. Profile README**
Aku buat repositori dengan username yang sama — repositori spesial yang muncul di profil GitHub. Isinya:
- Perkenalan singkat
- Tech stack utama (Flutter, Dart, Firebase, Riverpod)
- Link ke portfolio dan blog

Dulu profilku kosong — cuma foto dan jumlah repositori. Sekarang orang yang buka profilku langsung tahu: "Oh, ini Flutter developer."

**2. Satu repositori yang dirawat**
Alih-alih 10 proyek setengah jadi, aku fokus ke satu proyek: coding-skuy-apps. Aku rawat — bikin Readme yang rapi, pasang screenshot, tulis kontribusi guide. Dalam 3 bulan, star naik dari 3 ke 30+. Nggak banyak — tapi tiap star itu dari orang beneran yang lihat kualitasnya.

**3. Commit message yang rapi**
Dulu commit message-ku: "fix", "update", "asdf". Sekarang: "feat: add bookmark feature", "fix: handle empty search state", "refactor: extract widget into reusable component." Ini latihan kecil — tapi efeknya besar pas interview.

**4. Aktif berkontribusi**
Aku usahakan commit minimal 3-4x seminggu. Bukan karena dikejar — tapi karena GitHub contribution graph itu sinyal: "orang ini aktif ngoding, bukan cuma ngomong."

### Kisah Kedua: Saat GitHub Membuka Peluang

![GitHub pull request interface showing a code review conversation](/articles/images/journey/09-github-pull-request-review.jpeg)

Setelah setahun merawat GitHub, sesuatu yang tidak kusangka terjadi. Seorang CTO dari startup di Jakarta kirim DM di LinkedIn:

> *"Halo, lihat GitHub-mu. Kami lagi cari Flutter developer. Tertarik diskusi?"*

Dia lihat coding-skuy-apps di pinned repository. Dia baca Readme-nya. Dia cek commit history. Dia lihat cara aku nulis kode. Semua sebelum interview.

Waktu interview teknis, dia bilang: *"Kami udah lihat kode kamu — jadi technical interview-nya nggak perlu ngoding lagi. Kita bahas arsitektur aja."*

Bayangkan: **GitHub yang baik membuat rekruter tidak perlu tanya: "bisa ngoding?"** Karena jawabannya sudah jelas dari kode yang mereka lihat.

---

### Kisah Ketiga: Fork Juga Bisa Bernilai

Banyak orang kira fork itu tidak berguna. "Itu kan cuma copy paste repo orang." Tapi fork yang diikuti kontribusi berbeda.

Waktu aku fork flutter/flutter untuk berkontribusi — bukan cuma fork diam. Aku:
- Baca issue yang tagged "good first issue"
- Cari yang sesuai dengan kemampuanku
- Commit perbaikan
- Bikin pull request

Walaupun cuma 2 PR yang di-merge sepanjang tahun — itu cukup buat GitHub-ku punya jejak kontribusi ke proyek besar. Rekruter lihat dan berpikir: "Orang ini berkontribusi ke Flutter — berarti paham ecosystem-nya."

**Pelajaran:**
Fork lalu commit > fork diam > tidak fork sama sekali.

---

### Praktik yang Kupakai Sekarang

Ada 3 hal yang selalu kujaga di GitHub:

1. **Pinned repositories maksimal 3-6** — pilih yang terbaik, bukan yang terbaru. Satu proyek Flutter, satu kontribusi open source, satu blog/article.

2. **Readme yang informatif** — setiap repositori harus punya Readme yang jelaskan: apa ini, tech stack, screenshot, cara run. Tanpa Readme, repositori terlihat abandoned.

3. **Jaga aktivitas** — meskipun cuma minor update, usahakan ada aktivitas mingguan. Contribution graph yang hijau memberi sinyal positif.

### Jadi...

Kalau GitHub-mu sekarang masih isi fork dan proyek tutorial — itu tidak apa-apa. Semua mulai dari sana. Tapi **mulai hari ini, buat satu repositori orisinal**. Nggak perlu besar. Todo app boleh. Yang penting punyamu sendiri.

Dari situ, semuanya berubah.
