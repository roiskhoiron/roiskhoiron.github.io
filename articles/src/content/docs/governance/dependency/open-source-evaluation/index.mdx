# Open Source Evaluation

> *"Aku pake library Flutter yang kelihatan keren. 2 bulan kemudian — repository di-archive. Tidak ada maintainer."*

Pernah mengalami: pake library karena bintangnya banyak di GitHub. Ternyata 6 bulan tidak di-update. Tidak kompatibel dengan Flutter versi baru. Harus migrasi ke library lain.

Sejak itu, aku punya checklist sebelum pake library pihak ketiga.

---

### Kerangka Evaluasi

**1. Apakah ini terawat?**

Lihat:
- Kapan terakhir di-commit? (>6 bulan? Hati-hati.)
- Berapa banyak kontributor? (1 kontributor = bus factor tinggi)
- Apakah isu ditanggapi? (cek issues, apakah ada response dari maintainer)
- Apakah ada rilis terbaru? (versi yang stabil, bukan cuma commit)

**2. Apakah kualitasnya baik?**

- Ada test? (cek coverage. Kalau tidak ada test — risiko tinggi)
- API-nya intuitive? (dokumentasi — baca 5 menit. Kalau bingung, library mungkin tidak dirancang dengan baik)
- Dokumentasinya jelas? (contoh kode yang bisa di-copy paste)

**3. Bagaimana risikonya?**

- Lisensi — cocok dengan project? GPL untuk project komersial? Hati-hati.
- Dependency — library ini depend ke library lain. Semakin banyak dependency, semakin besar risiko konflik versi.
- Popularitas — bukan jaminan kualitas, tapi indikator. Library dengan 100 bintang risikonya lebih tinggi dari 10.000 bintang.

---

### Aturan Praktis

**Wajib dihindari:**
- Tidak ada commit >1 tahun
- Tidak ada test
- Satu maintainer

**Pertimbangkan ulang:**
- Dokumentasi minim
- API berubah-ubah tanpa migration guide
- Issue tidak pernah dijawab

> *"Setiap library yang kamu tambah adalah janji maintenance jangka panjang. Pikir dua kali."*
