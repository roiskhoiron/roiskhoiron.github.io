# Security Assessment

> *"Tahun lalu ada CVE di library yang kupakai. Untung tidak ekspos data user. Tapi itu peringatan."*

Setiap dependensi adalah potensi celah keamanan. Kamu tidak tahu siapa yang maintain library itu, apakah mereka jaga keamanan dengan baik.

---

### Yang Harus Dicek

**1. Riwayat CVE**

Cek: apakah library ini pernah punya celah keamanan? Gimana respon maintainer?

Tools:
- GitHub Security Advisories
- Snyk
- Dependabot (bawaan GitHub)

**2. Supply chain security**

- Apakah maintainer pakai 2FA? (kalau tidak, akun mereka bisa diretas)
- Apakah rilis ditandatangani? (verified)
- Apakah ada mekanisme reporting vulnerability?

**3. Dependency scope**

Semakin besar library, semakin besar permukaan serangan. Library yang butuh 50 dependency lain — risikonya lebih tinggi dari library yang zero dependency.

---

### Mitigasi

**Update rutin** — jangan biarkan library usang. Dependabot bisa otomatis bikin PR kalau ada update.

**Monitor CVE** — subscribe ke notifikasi keamanan untuk library yang kamu pake.

**Minimalkan dependensi** — setiap library yang ditambah adalah risiko tambahan. Tanya: apa memang perlu? Atau bisa pakai standar library?

**Pin versi** — jangan pake versi "latest" atau "^1.0.0" tanpa batas. Tentukan versi eksak, update setelah testing.

---

### Yang kulakukan

1. Dependabot aktif — setiap ada update keamanan, langsung notify
2. Review changelog sebelum update — jangan asal upgrade
3. Kalau library punya CVE kritis dan tidak di-fix — cari alternatif

> *"Keamanan dependensi bukan cuma tanggung jawab maintainer library — tapi tanggung jawab kamu yang memakainya."*
