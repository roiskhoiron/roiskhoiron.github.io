# Security Assessment

> *"Vulnerability pertama yang kuhadapi: library yang kupakai ternyata punya celah SQL injection. Beruntung, production belum terdampak."*

Waktu itu aku dapat notifikasi dari GitHub Dependabot: *"Critical vulnerability detected in library X."*

Jantungku berdetak cepat. Aku buka detailnya — ternyata versi library yang kupakai punya celah eksekusi kode jarak jauh.

Beruntung, celah itu belum dieksploitasi. Tapi sejak itu aku tidak pernah anggap remeh dependency security.

---

### Yang Perlu Dicek

**1. Known Vulnerability**

Tools otomatis yang kupakai:
- GitHub Dependabot (gratis)
- `npm audit` atau `dart pub outdated`
- Snyk (untuk yang lebih advance)

Jadwalkan pengecekan rutin. Bukan cuma sekali pas install.

**2. Reputasi Maintainer**

Cek:
- Apakah maintainer dikenal di komunitas?
- Apakah ada red flag di isu tracker?
- Apakah review keamanan pernah dilakukan?

**3. Supply Chain Risk**

Ini yang sekarang paling diperhatikan:
- Apakah maintainer cuma satu orang? (single point of failure)
- Apakah maintainer menggunakan 2FA?
- Apakah ada riwayat akun maintainer diretas?

---

### Praktik Harian

- **Lock version** — jangan pakai `^` untuk dependency kritis. Versi spesifik lebih aman
- **Update terjadwal** — jangan update semua dependency sekaligus. Update satu-satu, test, baru lanjut
- **Audit bulanan** — jadwalkan dependency audit tiap bulan
- **Minimalisasi** — makin sedikit dependency, makin sedikit permukaan serangan

---

### Pertanyaan Sebelum Install Library

*"Apakah saya bersedia bertanggung jawab atas keamanan library ini untuk 2 tahun ke depan?"*

Kalau jawabannya tidak, cari alternatif yang lebih terawat.

> *"Setiap dependency yang kamu tambah adalah pintu masuk potensial. Pastikan pintu itu terkunci."*
