# RFC (Request For Comments)

> *"Dulu aku pikir memutuskan sendiri itu efisien. Sampai keputusan yang salah harus dibayar oleh seluruh tim.""

RFC adalah kebalikan dari memutuskan sendiri. Ini undangan untuk diskusi sebelum keputusan besar diambil.

---

### Kenapa RFC?

Waktu di Papua Madhani Tech, ada satu keputusan arsitektur yang aku ambil sendiri. Pilih library A untuk state management.

Sepele, pikirku. Seminggu kemudian, engineer lain kesulitan karena library itu punya learning curve curam. Tim terhambat. Produktivitas turun.

Seandainya aku tulis RFC, mungkin ada yang bilang: *"Library B lebih ringan dan tim sudah familiar."*

---

### Kapan Pakai RFC?

| Situasi | RFC Perlu? |
|---------|-----------|
| Ganti framework utama | ✅ Wajib |
| Ubah arsitektur modul | ✅ Sangat disarankan |
| Pilih library baru | ⚠️ Tergantung dampak |
| Refactor internal | ❌ Tidak perlu |

---

### Format RFC Sederhana

```
Judul: [deskripsi singkat]

Masalah:
Apa yang ingin diselesaikan?

Usulan:
Bagaimana solusinya?

Alternatif:
Apa opsi lain? Kenapa tidak dipilih?

Dampak:
Apa efeknya ke sistem dan tim?
```

Cukup setengah halaman. Tidak perlu panjang.

---

### Proses RFC

1. **Tulis draft** — 15 menit cukup
2. **Share** — di Slack, PR, atau dokumen bersama
3. **Diskusi** — 1-3 hari, tergantung urgensi
4. **Keputusan** — tim atau tech lead yang putuskan
5. **Arsip** — simpan di folder RFC untuk referensi

---

### Tips

- Jangan terlalu lama. RFC yang dibahas >1 minggu biasanya tandanya tidak fokus
- Fokus ke masalah, bukan detail implementasi
- Ajak diskusi, bukan debat
- Kalau mentok, tech lead yang putuskan. Tim tetap jalan.

> *"RFC bukan birokrasi. Ini alat untuk memastikan semua orang bergerak ke arah yang sama."*
