# Security Review

> *"PR yang aku merge tanpa review keamanan ternyata mengandung celah. Beruntung ketahuan sebelum dieksploitasi."*

Itu pelajaran yang tidak akan kulupakan. Sebuah PR sederhana — tambah fitur upload gambar. Aku fokus review logic-nya. Lupa cek: apakah file yang diupload divalidasi tipenya?

Ternyata tidak. Attacker bisa upload file PHP, dan server bisa mengeksekusinya.

Sejak itu, security review jadi bagian wajib dari prosesku.

---

### Kapan Perlu Security Review?

Dari pengalaman, ini situasi yang paling rawan:

- **Fitur baru yang melibatkan data pengguna**
- **Perubahan di autentikasi/otorisasi** — ganti library auth, ubah role system
- **Dependency baru** — library eksternal mungkin punya celah
- **Sebelum rilis major** — kesempatan terakhir catch celah
- **Setelah incident** — cegah terulang

---

### Checklist Cepat

Sebelum merge, tanya:
- Apakah input divalidasi? (SQL injection? XSS?)
- Apakah ada data sensitif di response/log?
- Apakah ada credential hardcoded?
- Apakah library punya known vulnerability?
- Apakah konfigurasi aman? (debug mode di production?)

---

### Tools yang Kupakai

| Tools | Fungsi |
|-------|--------|
| Dependabot / Snyk | Cek vulnerability dependency |
| Linter | Deteksi potensi masalah keamanan |
| Code review manual | Lihat dengan mata kepala sendiri |

Tools membantu, tapi tidak menggantikan review manual. Terutama untuk logic validasi dan alur data.

---

### Prinsip

- Security review bukan blocking — ini safety net
- Kalau ragu, tanya tim. Jangan merge sendiri
- Dokumentasikan temuan. Tim belajar dari kesalahan

> *"Satu celah yang terlewat bisa berdampak ke ribuan user. Review adalah garis pertahanan terakhir."*
