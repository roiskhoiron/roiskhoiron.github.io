# Data Protection

> *"Waktu debugging, aku print object user ke console. Berisi email, nomor telepon, alamat. Lupa dihapus. Ke production."*

Beruntung, ada yang lihat di code review sebelum merge. Tapi itu momen yang bikin aku sadar: data protection bisa gagal karena hal sesederhana lupa hapus log.

---

### Yang Paling Sering Bocor

Dari pengalaman dan baca insiden orang lain:

| Sumber Kebocoran | Contoh |
|-----------------|--------|
| **Logging** | Print data sensitif ke console/log |
| **Error message** | Stack trace yang bocor ke client |
| **API response** | Field yang seharusnya tidak terekspos |
| **File upload** | File bisa diakses tanpa autentikasi |
| **Backup** | Backup database di public bucket |

---

### Klasifikasi Data

Aku bagi data jadi 3 level:

- **Publik** — nama produk, artikel — proteksi standar
- **Internal** — kode sumber, dokumentasi — akses terbatas
- **Sensitif** — password, token, data pengguna — enkripsi + akses ketat

Semakin sensitif, semakin ketat perlakuannya.

---

### Praktik Harian

1. **Jangan log data sensitif** — kalau perlu logging untuk debugging, pastikan dihapus sebelum production
2. **Enkripsi** — HTTPS untuk in transit, AES untuk at rest
3. **Minimalkan** — hanya kumpulkan data yang benar-benar dibutuhkan
4. **Hapus** — kalau data sudah tidak diperlukan, hapus
5. **Backup** — backup terenkripsi, akses terbatas

---

### Checklist

- [ ] Tidak ada data sensitif di log?
- [ ] Semua API pake HTTPS?
- [ ] Password di-hash?
- [ ] File upload terproteksi?
- [ ] Backup di tempat aman?

> *"Data pengguna bukan milikmu. Kamu cuma dititipi. Jaga seperti itu."*
