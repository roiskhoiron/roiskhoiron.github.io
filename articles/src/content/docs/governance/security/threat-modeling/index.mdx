# Threat Modeling

> *"Aku pikir aplikasiku aman. Sampai teman bilang: 'Coba klik ini...' dan aku bisa akses data orang lain."*

Threat modeling adalah latihan *"bagaimana jika"* sebelum serangan benar-benar terjadi. Lebih baik mencari celah sendiri daripada ditemukan orang jahat.

---

### Cara Sederhana: STRIDE

Aku pakai STRIDE untuk kategorisasi ancaman:

| Threat | Arti | Contoh |
|--------|------|--------|
| **S**poofing | Pura-pura jadi orang lain | Login pake credential orang |
| **T**ampering | Ubah data di tengah jalan | Intercept request, ganti nominal |
| **R**epudiation | Menyangkal sudah melakukan | "Saya tidak transfer uang itu" |
| **I**nformation Disclosure | Data bocor | Log response API yang terlalu detail |
| **D**enial of Service | Bikin sistem tidak bisa diakses | Request banjir ke server |
| **E**levation of Privilege | Dapat akses lebih dari seharusnya | User biasa jadi admin |

---

### Proses yang Kupakai

**1. Pilih aset paling berharga**

Data apa yang paling penting? (password, data keuangan, data pribadi)

**2. Pikirkan skenario terburuk**

Apa yang terburuk bisa terjadi pada aset ini? (Gunakan STRIDE)

**3. Cari mitigasi**

Apa yang bisa dilakukan untuk mencegah? Atau mengurangi dampak?

**4. Dokumentasikan**

Catat temuan. Jangan hanya di kepala.

---

### Contoh dari Pengalaman

Aplikasi: Sistem manajemen konten.

| Aset | Ancaman | Mitigasi |
|------|---------|----------|
| Data login | Spoofing (brute force) | Rate limiting, MFA |
| Artikel | Tampering (ubah konten) | Audit log, backup |
| API key | Information Disclosure (bocor di log) | Filter log, scan commit |

---

### Tips

- Libatkan tim, bukan hanya engineer. Perspektif berbeda menemukan celah berbeda
- Lakukan sejak desain, bukan setelah jadi
- Sederhana saja. Threat modeling 15 menit lebih baik daripada tidak sama sekali

> *"Cari celah sebelum orang jahat menemukannya."*
