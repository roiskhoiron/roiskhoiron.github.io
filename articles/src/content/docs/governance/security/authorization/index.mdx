# Authorization

> *"Aku ganti angka di URL dari `/user/5` jadi `/user/6`. Dan lihat data orang lain. Itu security hole."*

Autentikasi (siapa kamu) beda dengan otorisasi (apa yang boleh kamu lakukan). Banyak yang fokus ke auth (login, register) tapi lupa otorisasi.

Aku pernah: user A bisa lihat profil user B — tinggal ganti ID di parameter. Ini yang disebut **Broken Access Control**. Sangat umum, sangat berbahaya.

---

### Level Otorisasi

**1. Role-based (RBAC)**

User punya role: admin, editor, viewer. Tiap role punya akses berbeda.

Contoh: admin bisa hapus artikel. Editor bisa edit. Viewer cuma baca.

Implementasi: cek role user sebelum eksekusi. Kalau role tidak sesuai, return 403.

**2. Attribute-based (ABAC)**

Akses berdasarkan atribut: user, resource, environment.

Contoh: user hanya bisa edit artikel yang dia buat sendiri. Atau: artikel hanya bisa diedit jam kerja.

**3. Ownership-based**

User hanya bisa akses resource miliknya.

Ini yang paling sering dilanggar. API `/api/orders/` harus filter: hanya order milik user yang login. Bukan semua order.

---

### Common Mistakes

**1. Hanya cek auth, tidak cek otorisasi**

User sudah login → bisa akses apa pun. Salah. Login cuma bukti identitas, bukan izin akses.

**2. Otorisasi di client-side**

Tombol admin di-hidden dari non-admin. Tapi API-nya tetap bisa diakses. Attacker tinggal panggil langsung API — tidak perlu tombol.

Otorisasi harus di server-side. Client cuma UI.

**3. Insecure direct object reference (IDOR)**

API `/api/invoice/123` — siapa pun bisa akses dengan ganti angka. Padahal invoice itu milik user A, bukan user B.

Solusi: cek kepemilikan di server.

---

### Implementasi

Di CodingSkuy:
- Setiap endpoint punya cek: apakah user ini punya akses ke resource ini?
- Middleware yang handle otorisasi — tidak perlu diulang di tiap handler
- Default: deny. Kecuali explicitly allowed.

> *"Autentikasi tanpa otorisasi ibarat pintu terkunci — tapi semua ruangan di dalam tidak punya kunci."*
