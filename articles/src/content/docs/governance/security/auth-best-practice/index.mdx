# Authentication Best Practice

> *"Waktu aku mulai, autentikasi cuma bikin form login dan cek password pake ==. Syukurnya itu cuma proyek belajar.""

Aku mulai serius soal autentikasi ketika baca berita startup yang kena data breach karena password disimpan dalam plain text. Ribuan user terdampak.

Aku bertanya: *"Apakah aplikasiku aman dari serangan serupa?"*

Jawabannya: belum tentu.

---

### Yang Paling Mendasar

**1. Password**

Jangan pernah simpan password dalam bentuk asli. Pakai bcrypt atau argon2.

Awalnya aku pikir MD5 sudah cukup — cepat, simpel. Ternyata MD5 bisa di-crack dalam hitungan detik.

**2. JWT (Token)**

Yang kupelajari:
- Access token pendek umurnya (15 menit)
- Refresh token lebih panjang (7 hari) tapi disimpan aman
- Jangan simpan token di localStorage. Pakai httpOnly cookie
- Verifikasi signature di setiap request

**3. Rate Limiting**

Yang paling sering dilupakan. Tanpa rate limiting, attacker bisa coba jutaan password dalam semalam.

Aku terapin: 5 percobaan gagal = tunggu 15 menit.

---

### Checklist dari Pengalaman

- [ ] Password di-hash? (bcrypt/argon2, jangan MD5)
- [ ] Rate limiting di login?
- [ ] Session timeout setelah tidak aktif?
- [ ] Opsi MFA sudah ada? (minimal untuk admin)
- [ ] Log percobaan login?
- [ ] Token expire otomatis?

---

### Yang Kupelajari dari Kasus Nyata

- Jangan pernah percaya input user. Selalu validasi di server
- HTTPS wajib. Tidak ada alasan untuk HTTP di 2024
- Notifikasi login dari perangkat baru — fitur sederhana yang menyelamatkan banyak user

> *"Autentikasi adalah pintu depan aplikasi. Pastikan pintunya kuat."*
