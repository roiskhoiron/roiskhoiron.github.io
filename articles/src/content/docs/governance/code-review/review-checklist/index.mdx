# Review Checklist

> *"Dulu review cuma lihat 'apakah kodenya jalan.' Tapi banyak hal yang terlewat — security, error handling, performance."*

Awalnya review-ku tidak terstruktur. Baca kode, kasih komentar seadanya. Sampai ada bug yang lolos karena aku tidak cek error handling. Sejak itu, aku bikin checklist.

---

### Checklist Review

Setiap kali review, aku cek hal-hal ini:

**1. Logic & Correctness**

- Apakah kode menyelesaikan masalah yang dimaksud?
- Apakah ada edge case yang tidak dihandle?
- Apakah ada asumsi yang tidak tertulis?

Contoh: fungsi "getUser" — apa yang terjadi kalau user tidak ditemukan? Return null? Throw error? Return default?

**2. Error Handling**

- Apa yang terjadi kalau API error?
- Apa yang terjadi kalau input tidak valid?
- Pesan error-nya jelas?

Yang paling sering terlewat: error state. Kode untuk happy path biasanya bagus. Tapi error handling sering dilupakan.

**3. Security**

- Apakah input sudah divalidasi?
- Apakah ada SQL injection / XSS risk?
- Apakah data sensitif tidak bocor ke log?
- Apakah autentikasi dan otorisasi sudah benar?

**4. Performance**

- Apakah ada N+1 query?
- Apakah ada operasi berat di main thread?
- Apakah bisa di-cache?

**5. Maintainability**

- Apakah kodenya mudah dibaca?
- Apakah ada magic number / hardcoded value?
- Apakah test-nya sudah ditulis? (kalau ada)

---

### Cara Pakai

Checklist bukan alat untuk "menghakimi" kode — tapi untuk memastikan tidak ada yang terlewat. Tidak semua item harus dipenuhi setiap review. Tapi setidaknya, sudah dipertimbangkan.

> *"Checklist adalah pengingat, bukan aturan kaku. Gunakan untuk mengurangi human error, bukan untuk memperlambat."*
