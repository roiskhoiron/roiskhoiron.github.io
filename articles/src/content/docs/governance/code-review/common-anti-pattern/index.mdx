# Common Anti Pattern

> *"Anti-pattern adalah solusi yang terlihat benar di permukaan, tapi jadi mimpi buruk 3 bulan kemudian."*

Aku pelajari anti-pattern bukan dari buku. Tapi dari kode yang aku tulis sendiri, lalu aku sesali 6 bulan kemudian. Dan dari kode orang lain yang harus aku perbaiki.

Ini yang paling sering kutemui:

---

### 1. God Class — Satu Kelas untuk Semua

Dulu waktu belajar Android, semua logic kutulis di satu Activity. UI, logic bisnis, network call, database — semua di satu file.

Tanda-tandanya: file >500 baris, susah dibaca, error di satu tempat berdampak ke fitur lain.

**Pelajaran:** Pisah berdasarkan tanggung jawab. Testing jadi lebih mudah. Bug lebih gampang dilacak.

### 2. Copy-Paste Programming

Pernah menemukan bug di satu fitur, lalu sadar bug yang sama ada di 3 file lain karena kodenya di-copy-paste?

Aku alami itu. Waktu itu aku buru-buru, copas kode dari fitur A ke fitur B, ubah sedikit. Hemat 5 menit. Tapi ketika fitur A diperbaiki, fitur B lupa diperbaiki. Bug muncul di production.

**Pelajaran:** Kalau kode perlu dipakai di banyak tempat, ekstrak ke fungsi/kelas terpisah.

### 3. Golden Hammer — Pakai Satu Tools untuk Semua

Dulu aku sempat kecanduan NoSQL. "Semua pake MongoDB." Sampai aku bangun sistem yang butuh relasi kompleks. Pelan. Berantakan. Akhirnya migrasi ke PostgreSQL.

**Pelajaran:** Pilih tools berdasarkan masalah, bukan karena lagi tren.

### 4. Premature Optimization

Aku pernah optimasi query yang belum lambat. Bikin cache di semua layer. Kode jadi kompleks. Ternyata setelah diukur, bottleneck-nya di tempat lain.

**Pelajaran:** Buat sederhana dulu. Ukur. Baru optimasi yang terbukti lambat.

### 5. Fear of Removing Code

Ada file dengan komentar *"TODO: hapus kalau sudah tidak dipakai"* yang bertahan 2 tahun. Karena takut dihapus, padahal tidak ada yang memanggil.

**Pelajaran:** Git menyimpan sejarah. Kalau tidak dipakai, hapus.

---

> *"Anti-pattern bukan dosa. Tapi pelajaran yang kamu bayar dengan waktu debugging."*
