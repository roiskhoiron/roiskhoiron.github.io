# Code Review Anti Pattern

> *"Pernah review cuma bilang 'LGTM' tanpa lihat kode. Besoknya error di production. Itu salahku."*

Code review — ritual yang sering jadi formalitas. Padahal kalau dilakukan dengan benar, ini safety net paling efektif.

Aku pernah di kedua sisi: yang kena review asal-asalan, dan yang jadi reviewer malas.

---

### Anti-pattern yang Sering Terjadi

**1. LGTM tanpa baca**

Lihat PR, scroll, komentar "LGTM!" — tapi tidak membaca kode sama sekali. Ini berbahaya. Error kecil lolos, security issue terlewat.

Kenapa orang lakuin ini? Karena buru-buru, atau merasa "ah, ini developer senior, pasti bener."

**2. Review terlalu lambat**

PR didiamkan berhari-hari. Konteks hilang. Developer harus refresh ingatan. Kadang terpaksa merge tanpa review karena deadline.

Di satu proyek, PR saya nunggu 2 minggu untuk direview. Pas direview, saya sudah lupa detailnya. Harus baca ulang kode dari awal.

**3. Nitpicking berlebihan**

Reviewer fokus ke spasi, penamaan variabel, dan formatting. Tapi tidak lihat logic error atau security issue.

Padahal formatting bisa diurus oleh linter dan formatter otomatis. Waktu review seharusnya untuk hal yang tidak bisa dideteksi mesin.

**4. Ego trip**

Reviewer merasa kodenya selalu lebih baik. Semua PR dikomentari dengan "harusnya pake cara saya."

Padahal coding itu personal. Selama ikut standar tim dan logikanya benar, banyak cara yang valid.

---

### Cara Menghindari

- **Prioritaskan review** — review dalam 24 jam, maksimal 48 jam
- **Baca dengan teliti** — jangan LGTM tanpa baca. Kalau tidak sempat, bilang saja
- **Fokus ke logic, bukan style** — biarkan linter urus formatting
- **Tanya, bukan perintah** — "Kenapa pake pendekatan ini?" lebih baik dari "Ganti pake cara saya"

> *"Code review yang baik bukan mencari kesalahan — tapi belajar bersama untuk menghasilkan kode yang lebih baik."*
