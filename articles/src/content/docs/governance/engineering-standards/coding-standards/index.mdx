# Coding Standards

> *"File pertama pake 2 spasi. File kedua pake 4 spasi. File ketiga pake tab. Semua di proyek yang sama."*

Pernah lihat kode yang formatting-nya tidak konsisten dalam satu proyek? Aku pernah. Dan itu menyebalkan. Setiap buka file baru, harus adaptasi gaya penulisan yang berbeda.

Coding standards bukan soal "cara yang benar" — tapi soal **konsistensi**. Tidak masalah pake 2 spasi atau 4 spasi — yang penting sama di semua file.

---

### Area yang Perlu Distandarisasi

**1. Formatting**

- Spasi vs tab — pilih salah satu untuk semua
- Panjang baris — 80 atau 120 karakter?
- Kurung kurawal — di baris yang sama atau baris baru?

Semua ini bisa diurus oleh formatter otomatis (dart format, prettier, dll). Jangan buang waktu debat formatting — set formatter, jalanin, selesai.

**2. Naming conventions**

- camelCase, snake_case, atau PascalCase? Pilih dan konsisten.
- Singkatan — boleh atau tidak? "btn" atau "button"? "ctx" atau "context"?

**3. File organization**

- Satu class per file? Atau beberapa class kecil dalam satu file?
- Naming file — sesuai class name?

---

### Tools yang Kupakai

**Linter** — menangkap kesalahan standar secara otomatis. Di Flutter pake `flutter analyze` atau `dart analyze`.

**Formatter** — merapikan kode secara otomatis. `dart format` untuk Flutter/Dart. `prettier` untuk JavaScript/TypeScript.

**Editor config** — file `.editorconfig` memastikan semua editor punya setting yang sama. Spasi vs tab, indent size, line ending.

---

### Tips

- Jangan debat formatting di code review — itu tugas formatter
- Standar ditulis dan disepakati — tidak boleh retoraktif
- Update standar secara periodik — teknologi berubah
- Automation lebih baik dari manual — makin sedikit yang manual, makin konsisten

> *"Standar coding bukan untuk membatasi kreativitas. Tapi untuk menghilangkan debat tidak produktif tentang hal yang tidak penting."*
