# Architecture Guidelines

> *"Satu tim pake BLoC, satu lagi pake Riverpod. Satu pake layer-first, satu pake feature-first. Hasilnya: kode tidak konsisten."*

Di satu proyek, setiap developer punya gaya arsitektur sendiri. Hasilnya: kode yang tidak konsisten. Pindah dari modul A ke B, pola berubah total. Butuh adaptasi tiap kali.

Architecture guidelines adalah solusinya — bukan untuk mengekang, tapi untuk menyepakati standar.

---

### Komponen Guidelines

**1. Struktur folder**

Di mana taruh model? Di mana taruh screen? Bagaimana organize fitur?

Keputusan: feature-first atau layer-first? Folder `core/` atau `shared/`? Naming conventions untuk folder.

Aku pake feature-first: setiap fitur punya folder sendiri. Tapi `core/` untuk shared code.

**2. State management pattern**

Pola apa yang dipakai? Kapan pake setState, kapan pake Riverpod?

Guidelines-ku:
- Data lokal → setState
- Data shared antar screen → Riverpod
- Real-time → BLoC

**3. Dependency injection**

Manual atau pake package? Bagaimana cara inject dependency?

Aku pake Riverpod (ref) atau constructor injection. Tidak pake get_it.

**4. Error handling**

Gimana handle error? Return type? Global error handler?

Guidelines: Semua fungsi yang bisa error return Result type (Success/Failure). Ada global handler untuk unhandled error.

---

### Panduan Sederhana

| Hal | Batasan |
|-----|---------|
| Baris per file | &lt;300 baris |
| Baris per fungsi | &lt;30 baris |
| Method publik per kelas | &lt;5 |
| Parameter per fungsi | &lt;3 |

Angka ini bukan aturan kaku — tapi panduan. Kalau melebihi, mungkin waktunya refactor.

---

### Cara Menerapkan

1. Tulis guidelines — 1-2 halaman
2. Diskusikan dengan tim — semua harus setuju
3. Terapkan via code review dan linter (otomatis)
4. Review secara periodik — update kalau tidak relevan

> *"Konsistensi lebih penting dari kesempurnaan."*
