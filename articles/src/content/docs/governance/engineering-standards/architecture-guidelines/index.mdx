# Architecture Guidelines

> *"Dulu setiap folder punya struktur sendiri-sendiri. Tergantung mood programmernya. Waktu aku harus cari file, butuh 10 menit."*

Itu pengalaman di proyek pertama. Tiga engineer, tiga gaya arsitektur. Satu pake MVC, satu pake spaghetti, satu campur aduk.

Ketika aku harus memperbaiki bug di modul yang bukan aku tulis, waktu habis untuk memahami struktur, bukan untuk memperbaiki bug.

---

### Kenapa Guidelines Diperlukan

Bukan untuk membatasi kreativitas. Tapi untuk:
- Engineer baru bisa produktif dalam hari, bukan minggu
- Bug lebih cepat ditemukan karena struktur konsisten
- Refactor lebih aman karena kode terprediksi

---

### Prinsip yang Kupakai

**1. Separation of Concerns**

Setiap layer punya tugas jelas. Presentasi urus UI. Domain urus logic. Data urus penyimpanan. Tidak campur aduk.

Aku belajar ini setelah pindah dari Android Native (semua di Activity) ke Flutter (architectur jelas).

**2. Dependency Rule**

Domain tidak tahu soal framework atau database. Ini yang membuat testing jadi mudah.

**3. Testability**

Kalau komponen susah di-test, biasanya ada yang salah dengan arsitekturnya.

---

### Struktur Folder Standar

```
lib/
├── core/          # utilitas bersama
├── fitur_a/
│   ├── data/      # repository, data source
│   ├── domain/    # use case, entity
│   └── ui/        # halaman, widget
├── fitur_b/
│   └── ...        # sama
└── main.dart
```

---

### Panduan Sederhana

| Hal | Batasan |
|-----|---------|
| Baris per file | &lt;300 baris |
| Baris per fungsi | &lt;30 baris |
| Method publik per kelas | &lt;5 |
| Parameter per fungsi | &lt;3 |

> *"Konsistensi lebih penting dari kesempurnaan."*
